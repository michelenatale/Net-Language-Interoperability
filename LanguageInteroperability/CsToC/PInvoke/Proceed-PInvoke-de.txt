Für den Buildvorgang der "cadd.c" Datei muss folgendermassen vorgegangen werden.

Sicherstellen, dass die PLattform "x64" ist.

Command Promt öffen, es muss jedoch der x64 native sein.
>> also Windows-Startmenu und "x64 Native Tools" eingeben, dann sollte der richtige kommen.

Sobald "x64 Native Tools Command Prompt for VS 2026" gestartet ist, den pfad zur cadd.c setzen mit cd..
>> Laufwerk wechsel mit X:
>> cd X:\[..]\NetLanguageInteroperability\LanguageInteroperability\CsToC\PInvoke

Mit "dumpbin /headers cadd.dll | findstr machine" 
>> kann geprüft werden um was für ein "Command Promt es sich handelt"

Mit "cl /LD cadd.c /Fe:cadd.dll"
>> die cadd.c wird kopiliert und es wird eine cadd.dll generiert.

Auf die Frage was die cadd.c für ein build ist, da kann nur geantwortet werden, dass der cl-Befehle "release" nicht so kennt wie VS2026. 
Es handelt sich aber schon um eine quasi optimiertes Build, und daher darf man sagen, dass es um ein release-ähnliches build.

Sofern wirklich eine VS2026-Realease-Build verlangt wird kann auch der Befehl
"cl /LD /O2 /GL /Gy /DNDEBUG cadd.c /Fe:cadd.dll"
verwendet werden, Das entspricht ziemlich genau einem Visual‑Studio‑Release‑Build.

******** ******** ******** ******** ******** ******** ******** ******** 
******** ******** ******** ******** ******** ******** ******** ******** 

Es gäbe den Befehl "cl /LD /O2 cadd.c /Fe:cadd.dll" der würde noch mehr otimieren, ist aber in den meisten Fällen nicht notwendig, es ist aer auch nicht falsch wenn es gemacht wird.
>> /O2 = maximale Optimierung (entspricht ziemlich Release)

Für einen Debug-ähnlichen Build 
"cl /LD /Zi /Od cadd.c /Fe:cadd.dll"
ist jedoch eigentlich nicht notwendig, aber auch nicht falsch.
